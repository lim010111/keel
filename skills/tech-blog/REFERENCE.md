# 기술 블로그 작성 — 상세 참고

## 글 구조 템플릿

```
# 제목
   내용을 그대로 요약한 제목. 과장·낚시 없이.
   예) O: "Spring Boot에서 @Transactional 전파 옵션 정리"
       X: "@Transactional, 이것만 알면 끝!"

## 인사말 (1~2문장)
   가벼운 인사 + 이 글에서 다룰 주제 한 줄. 길게 끌지 않는다.

## 도입
   이 글이 다루는 범위와, 왜 다루는지(어떤 문제/상황에서 필요한지).
   3~5문장. 배경 설명이 길어지면 본문으로 넘긴다.

## 본문 (헤딩으로 구획)
   개념 → 사용법 → 예시 순서가 기본. 주제에 맞게 조정한다.
   - 코드 블록: 언어 지정(```java, ```bash 등). 실제 동작하는 코드만.
   - 표: 옵션 비교, 버전별 차이 등 나열형 정보에 사용.
   - 이미지/다이어그램: 글로 설명하기 어려운 구조에만. 캡션을 단다.
   - 버전·날짜·API 이름은 정확히 적는다.

## 주의사항·제한사항 (해당될 때만)
   알려진 버그, 적용 안 되는 환경, 흔한 실수 등.

## 정리
   본문의 핵심을 짧게 다시 짚는다. 여기서 새 정보를 추가하지 않는다.

## 참고 자료
   본문에서 인용한 출처를 링크 목록으로 정리한다.
```

## 쓰지 않는 표현

| 분류 | 피할 표현 |
|------|-----------|
| 어그로 | 충격, 경악, ~의 비밀, 이것만 알면, 무조건, 완벽 정리, 끝판왕, 총정리, 안 보면 후회 |
| 과장 | 혁명적, 게임 체인저, 압도적, 미친, 엄청난, 최고의, 단언컨대, 정말 강력한 |
| 추측 | 아마도, ~인 듯하다, ~일 것이다, 근거 없는 "보통은/대부분" |
| 공허한 강조 | 의미 없이 붙는 매우·정말·굉장히 같은 부사 |

### 고쳐쓰기 예시

- X: "이 기능은 정말 혁명적으로 개발 속도를 높여줍니다."
  O: "이 기능을 쓰면 빌드 설정 파일을 직접 작성하지 않아도 된다."
- X: "아마 내부적으로 캐시를 사용할 것입니다."
  O: "(출처 확인 후) 공식 문서에 따르면 내부적으로 LRU 캐시를 사용한다." 또는 해당 문장 삭제.
- X: "@Transactional만 붙이면 모든 게 해결됩니다!"
  O: "@Transactional을 붙이면 메서드 실행이 하나의 트랜잭션으로 묶인다."

## 출처 표기

- 외부 사실(공식 문서, 릴리스 노트, 벤치마크 등)을 언급할 때는 본문에 출처를 밝히거나, 글 끝 "참고 자료"에 링크를 남긴다.
- 버전 번호, 날짜, 함수/옵션 이름은 출처에 적힌 그대로 옮긴다. 기억에 의존해 적지 않는다.
- 코드 예시는 가능하면 실제로 실행해 확인한다. 확인하지 못했으면 그 사실을 짧게 밝힌다.

## 자가 검수 체크리스트

- [ ] 모든 기술적 주장에 근거가 있는가 (코드 / 출처 / 사용자 확인)
- [ ] 추측으로 빈틈을 메운 문장이 없는가
- [ ] 제목·본문에 과장·어그로 표현이 없는가
- [ ] 의견은 의견임을 밝히고 근거를 댔는가
- [ ] 정보를 주지 않는 군더더기 문장을 모두 제거했는가
- [ ] 코드 예시가 동작하는가 (확인 못 했으면 명시했는가)
- [ ] 버전·날짜·이름이 출처와 일치하는가
- [ ] 인용한 출처가 모두 "참고 자료"에 있고 링크가 유효한가
